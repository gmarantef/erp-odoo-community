# Error Solving

Principios para abordar errores encontrados durante el desarrollo.

## Reglas

1. **Resolver antes de avanzar.** Todo error encontrado debe resolverse antes de continuar con el desarrollo o implementación de siguientes pasos.

2. **Simplicidad primero.** La solución debe ser la más simple posible que resuelva el problema. Si la solución parece compleja, es señal de que no se ha encontrado la causa raíz.

3. **Sin sobreingeniería.** No añadir capas de complejidad (scripts, wrappers, workarounds) para esquivar un error. Si hay que rodear el problema en lugar de resolverlo, la solución es incorrecta.

4. **Documentación oficial como punto de partida.** Los errores suelen tener una causa simple documentada. Antes de buscar en foros o tutoriales, consultar la documentación oficial de la herramienta que falla.

## Flujo de resolución

```
1. Leer el error completo — no interpretar a medias
2. Identificar la causa raíz (no el síntoma)
3. Consultar documentación oficial de la herramienta implicada
4. Proponer la solución más simple que ataque la raíz
5. Validar que el error desaparece antes de continuar
```
